## 计划：项目完整性与体验提升 (Completeness & Polish)

基于当前项目的核心业务功能（进销存部分：商品/库存/供应商/采购，零售部分：收银收单/会员/积分/权限）已基本完成闭环。本阶段将专注于**不新增大型扩展模块**的前提下，重点补齐系统作为开源或商用标准 POS 的基础完整性、健壮性与交互体验。

### 1. UI / UX 与视觉体验
- [ ] **暗色模式（Dark Mode）（历史遗留遗留）**
  - 落实前期 `PLAN.md` 规划的 Tailwind `dark:` 换肤。
  - 支持跟随系统与手动覆盖（侧边栏切换按钮）。
- [ ] **平板与触控适配优化 (Tablet / POS 级要求)**
  - POS 系统常在 iPad/平板/触摸屏终端上使用，重点审查收银台 (`dashboard.vue`)、结账弹窗区的按钮触控面积与横竖屏响应式布局。
- [ ] **加载状态与防重复提交**
  - 排查所有表单提交（创建订单、一键入库）及核心列表加载。
  - 确保均具备 `Skeleton` 或 `LoaderCircle` 过渡，并给按钮加上 `submitting` 锁，防止网络卡顿时操作员连续点击导致重复下单/重复入库。

### 2. 后端健壮性与数据安全审查
- [ ] **强事务保障 (Transactions Review)**
  - 审查 `order-service.ts` 和 `purchase-service.ts`、`order-refund-service.ts`。
  - 确保“新增订单 + 扣除库存 + 扣除/增加积分 + 记录审计日志”这一类复合操作必须在同一个 Prisma 事务 `$transaction` 中，避免数据不一致。
- [ ] **防呆与边界值校验**
  - 检查服务端接口的参数校验，杜绝出现诸如（负数金额、负数采购数量、折扣率大于100%）的情况入库。
- [ ] **资源关联删除保护**
  - 确保有关联订单的客户、有关联采购的供应商、已有流水的产品不可直接物理删除，只能被“停用”或进行外键约束校验拦截。

### 3. 运维、运维工具与基础导出
- [ ] **数据账单导出**
  - 虽然不额外开发复杂图表，但作为 POS 系统，至少需要为“订单流水 (`orders.vue`)”、“采购单 (`purchase-orders.vue`)” 和 “库存明细” 加入原生的无样式 CSV 导出/下载功能，以供外部对账。
- [ ] **部署文件齐备**
  - 梳理并提供标准化的 PM2 配置 (如 `ecosystem.config.cjs`) 和环境变量说明文档 (`.env.example`)，以满足标准的 Node.js / Nuxt 3 生产上线要求。
