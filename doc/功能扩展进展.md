## 审计日志排序异常修复 (2026-04-29)

### 🐞 问题现象
操作日志页面未显示今天（4/29）的日志，最新记录停留在 4/27。

### 🔍 根因分析
`AuditLog` 表的 `createdAt` 字段存在 **两种数据类型混存**：

| 数据批次 | 存储类型 | 产生原因 |
|---------|---------|---------|
| 4/27 早期（28条） | `text`（ISO 字符串） | `schema-bootstrap.ts` 建表时使用 `DEFAULT CURRENT_TIMESTAMP`，SQLite 以文本写入 |
| 4/27 晚期及之后（20条） | `integer`（Unix 毫秒） | 重构后 `prisma.auditLog.create()` 使用 Prisma 标准序列化 |

SQLite 对混合类型列做 `ORDER BY DESC` 时，`text` 类型按字典序始终排在 `integer` 之前（因字母 ASCII > 数字），导致今天的新日志全部被压到列表尾部、无法在第一页显示。

### ✅ 修复方式
编写迁移脚本，将全部 28 条 `text` 类型的 `createdAt` 统一转换为 `integer`（Unix 毫秒时间戳），与 Prisma ORM 的序列化标准对齐。修复后排序恢复正常。

- **动态库存预测**：不再依赖静态的“库存<10预警”，而是通过让大模型分析近 30 天销量趋势，预测未来一周可能缺货的商品。
- **智能审计与异常拦截**：定时基于 `AuditLog` 分析操作行为（如：特定时间段的高频退款、异常登录等），生成安全预警简报。

### 核心落地方法论：
1. **结构化数据提供**：从单纯输入文本，向传入 JSON 格式的时序数据/订单聚合数据转变。
2. **Function Calling (工具调用)**：使 LLM 输出特定格式动作（如直接返回补货清单 API 所需的 JSON），让 AI 具备“执行”能力。
3. **流式响应 (Streaming)**：为提升体验，AI 交互尽量采用 SSE (Server-Sent Events) 流式输出。
